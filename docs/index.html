Contenuti

Sprint 0: analisi dei requisiti, descrizione e modellazione dei componenti, piano di test e goal per Sprint 1.

Requisiti

A Maritime Cargo shipping company (fron now on, simply company) intends to automate the operations of load of containers in the ship's cargo hold (or simply hold). To this end, the company plans to employ a Differential Drive Robot (from now, called cargorobot). The hold is a rectangular, flat area with an Input/Output port (IOPort). The area provides 4 slots to store the containers and a slot named slot5.
In the picture below:
  • The slots1-4 depict the hold areas reserved to store one container each
  • The slot5 depicts an area, where the cargorobot must temporarily store a container, before to place it in one of the slots1-4. During temporary storage, a 'marker' device labels the container with an identification barcode and signals when this marking activity is completed
  • The IOPort is a device with a pushbutton and a display. The pushbutton is pressed by the customer in order to to send a request to load a container on the cargo. The display is used to show the answer to the request and to show the current state of the hold
  • The sensor associated to the IOPort is a device (a sonar) used to detect the presence of a container, when it measures a distance D, such that D < DFREE/2, during a reasonable time (e.g. 3 secs)

Analisi del Problema

Hold

La hold è lo spazio di lavoro del cargorobot. Usiamo un sistema di coordinate basato sulla dimensione del robot: definiamolo come cerchio con raggio r, la stanza di base B e altezza H. 

int hold[ ][ ]
int width  = B/2r        // width*2r  < B < width*(2r+1)
int height = H/2r        // height*2r < H < height*(2r+1)

La posizione HOME è definita come hold[0][0]. Il committente fornisce la mappa con i seguenti punti di interesse: 

hold[1][2] = slot1
hold[1][3] = slot2
hold[3][2] = slot3
hold[3][3] = slot4
hold[2][4] = slot5           // slot temporaneo per marcatura
hold[0][5] = IOPORT     // posizione della IOPort nella griglia
Natura software: la hold è modellata come POJO, in quanto entità passiva che contiene altri POJO. La logica di orchestrazione risiede nel cargoservice.

Slot

Gli slot sono celle della hold che contengono al massimo 1 container. Distinguiamo due tipologie:

  • slot1-4: slot permanenti. Destinazione finale del container.
  • slot5: slot transitorio. Il container vi sosta per la marcatura (barcode). Un dispositivo "marker" applica l'etichetta e segnala il completamento al sistema.

Modelliamo gli slot come POJO:

bool occupato       // false = libero, true = container presente
Natura software: gli slot sono modellati come POJO in quanto entità passive.

IOPort

La IOPort è il punto di ingresso/uscita dei container nella hold.

Logicamente, la IOPort è composta da:

  • pushbutton: pulsante fisico. Premuto dal cliente, invia una richiesta di carico al cargoservice.
  • display: mostra risposta e stato della hold. Nel sistema corrisponde alla webGUI (componente fornito).
Natura software: dal punto di vista fisico, è un POJO collocato nella griglia alla posizione hold[0][5].
Al livello logico invece, pushbutton e display sono ATTORI

Sonar

Il sonar è un componente attivo che rileva la presenza o l'assenza di un container presso la IOPort, misurando una distanza D.
Data una distanza DFREE, valgono le seguenti proprietà: 

D < DFREE/2                           = container presente, IOPort occupata
DFREE/2 <= D <= DFREE           = misurazione valida, IOPort libera
D > DFREE per più di 3 sec     = sistema entra in Out-of-Service

Emette l'evento:

sonardata : sonar(DISTANCE)    // evento periodico con distanza misurata
Natura software modellato come ATTORE in quanto componente attivo.

Led

Il led è un dispositivo di notifica visiva. I suoi comportamenti:

  • BLINK: stato ENGAGED
  • OFF: stati IDLE / DISENGAGED
Natura software modellato come ATTORE in quanto componente attivo che risiede su un nodo remoto, ad esempio Raspberry Pi.

Il core business

Il cargoservice è il macro-componente che implementa il core-business e Orchestra tutti gli altri componenti, modellato quindi come ATTORE.
La logica di business principale è composta da:

  • dove e quando spostare il cargorobot
  • accettare o rifiutare una richiesta di carico
  • quale slot libero associare alla richiesta che si sta servendo
  • mandare out of order il sistema quando si accorge di un malfunzionamento del sonar

Test Plans

I test sono scritti in Java (JUnit). I POJO di supporto (hold, slot, etc...) sono fittizzi e servono ad avere dei test iniziali, per poi concretizzarli con gli oggetti reali.
Assumiamo che prima di ogni test il sistema sia avviato e il cargoservice sia raggiungibile tramite conn (connessione TCP alla porta del contesto QAK).

T01: Richiesta accettata

Precondizione: sistema in IDLE, almeno uno slot1-4 libero, IOPort libera.
Atteso: cargoservice risponde con il nome dello slot riservato; sistema passa a ENGAGED. 
@Test
public void testTF01_LoadRequestAccepted() throws Exception {
    // Simula pressione del pushbutton: invia richiesta di carico
    String request = CommUtils.buildRequest("tester",
            "load_request", "load_request(client1)",
            "cargoservice").toString();

    String response = conn.request(request);
    System.out.println("TF01 response: " + response);

    // Il cargoservice deve rispondere con il nome dello slot riservato
    assertTrue("TF01: richiesta accettata con slot assegnato",
            response.contains("load_accepted"));
}

T02: Container depositato in tempo

Precondizione: sistema in ENGAGED (TF01 già eseguito).
Atteso: il sonar rileva il container, il cargorobot esegue il ciclo completo, il sistema torna a IDLE. 
@Test
public void testTF02_ContainerDepositedInTime() throws Exception {
    // Prima: ottieni lo stato ENGAGED
    String req = CommUtils.buildRequest("tester",
            "load_request", "load_request(client1)", "cargoservice").toString();
    conn.request(req);

    // Simula: il sonar rileva il container (stub emette sonardata con D < DFREE/2)
    CommUtils.forwardOnInterconn(conn, CommUtils.buildDispatch("tester",
            "sonar_stub_trigger", "sonar_stub_trigger(near)", "sonarstub").toString());
    CommUtils.delay(5000); // attende il ciclo di movimentazione completo

    // Verifica che il sistema sia tornato a IDLE (slot aggiornato)
    String statusReq = CommUtils.buildRequest("tester",
            "get_hold_state", "get_hold_state(X)", "cargoservice").toString();
    String status = conn.request(statusReq);
    System.out.println("TF02 hold state: " + status);

    assertTrue("TF02: sistema tornato a IDLE, slot occupato",
            status.contains("slot_occupied"));
}

T03: Timeout deposito (DISENGAGED)

Precondizione: sistema in ENGAGED.
Atteso: allo scadere dei 30 sec senza rilevazione, il sistema torna a IDLE con lo slot liberato. 
@Test
public void testTF03_TimeoutDisengaged() throws Exception {
    // Ottieni stato ENGAGED
    String req = CommUtils.buildRequest("tester",
            "load_request", "load_request(client1)", "cargoservice").toString();
    String resp = conn.request(req);
    assertTrue("TF03 setup: richiesta accettata", resp.contains("load_accepted"));

    // Attende il timeout (configurare a 5 sec in test mode)
    CommUtils.delay(6000);

    // Lo slot deve essere stato liberato: una seconda richiesta deve essere accettata
    String req2 = CommUtils.buildRequest("tester",
            "load_request", "load_request(client2)", "cargoservice").toString();
    String resp2 = conn.request(req2);
    System.out.println("TF03 second request: " + resp2);

    assertTrue("TF03: sistema tornato a IDLE dopo timeout (slot liberato)",
            resp2.contains("load_accepted"));
}

T04: Hold piena

Precondizione: tutti gli slot1-4 occupati.
Atteso: cargoservice rifiuta la richiesta. 
@Test
public void testTF04_HoldFull() throws Exception {
    // Setup: forza tutti gli slot occupati tramite dispatch diretto
    for (int i = 1; i <= 4; i++) {
        CommUtils.forwardOnInterconn(conn, CommUtils.buildDispatch("tester",
                "set_slot_occupied", "set_slot_occupied(" + i + ")", "cargoservice").toString());
    }
    CommUtils.delay(200);

    String request = CommUtils.buildRequest("tester",
            "load_request", "load_request(client1)", "cargoservice").toString();
    String response = conn.request(request);
    System.out.println("TF04 response: " + response);

    assertTrue("TF04: richiesta rifiutata perché hold piena",
            response.contains("load_rejected"));
}

T05: IOPort occupata

Precondizione: sonar misura D < DFREE/2 (container già in IOPort).
Atteso: cargoservice risponde retrylater. 
@Test
public void testTF05_IOPortOccupied() throws Exception {
    // Simula IOPort occupata: stub sonar emette distanza vicina
    CommUtils.forwardOnInterconn(conn, CommUtils.buildDispatch("tester",
            "sonar_stub_trigger", "sonar_stub_trigger(near)", "sonarstub").toString());
    CommUtils.delay(500);

    String request = CommUtils.buildRequest("tester",
            "load_request", "load_request(client1)", "cargoservice").toString();
    String response = conn.request(request);
    System.out.println("TF05 response: " + response);

    assertTrue("TF05: risposta retrylater perché IOPort occupata",
            response.contains("retrylater"));
}

Goal Sprint 1

Lo Sprint 1 ha come obiettivo la realizzazione del nucleo operativo del sistema, senza dipendenze da hardware fisico (sonar reale, pushbutton fisico, led fisico).

Sottoinsieme di Requisiti (goal Sprint 1)

  • Implementare cargoservice con stati: IDLE, ENGAGED, DISENGAGED (OUT_OF_SERVICE rimandato a Sprint 2, con sonar reale)
  • Implementare robotservice come wrapper per robotsmart26
  • Gestire la riserva degli slot e il ciclo di movimentazione
  • Gestire il timeout di 30 secondi per il deposito
  • Il sonar è simulato (stub programmabile)
  • Hold e slot: strutture dati in-memory (POJO)

Test da superare in Sprint 1

  • TF01: Richiesta accettata
  • TF02: Container depositato in tempo
  • TF03: Timeout / DISENGAGED
  • TF04: Hold piena
  • TF05: IOPort occupata

Componenti prodotti in Sprint 1

  • cargoservice.qak: attore QAK, core business
  • robotservice.qak: attore QAK, wrapper robotsmart26
  • CargoServiceTest.java: test JUnit (TF01-TF05)
