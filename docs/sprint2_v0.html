Analisi del Problema

sensor

Partiamo dall'analisi del problema del componente sensor, che attraverso un sensore a ultrasuoni, deve essere in grado di rilevare la distanza di frotne a io port... ... Da requisiti, il tipico flusso di lavoro del sonar è il seguente:

  1. Il sonar esegue una rilevazione della distanza D
  2. A seconda di D, è necessario interpretare la distanza come:
    • D < DFREE/2: container presente presso la IOPort
    • DFREE/2 < D < DFREE: IOPort libera
    • D > DFREE: sensore guasto
    3. ANALISI: il sensore fisico è stato fortnito dal committente, con richiesta di essere montato su un dispositvo Raspberry. Inoltre, ci vengono fortnite le librerie python per gestirne il funzionamento, ottenendo le rilevazioni per poi elaborarle. Per gestire la distanza come da reuisiti bisogna trasformare queste rilevazioni in dati utili al nostro sistema, quindi bisogna decidere quale componente ha l'onere di questa elaborazione. Abbiamo deciso di creare un componente chiamato sensorservice, che si trova sullo stesso nodo del sensore fisico, che ha il compito di ricevere i dati grezzi inviati dal sensore e inviare i mesaggi necessari a cargoservice ... etc ...
  3. Se il sistema entra in OOS, ne uscirà alla prima misurazione D < DFREE
(LINK PYTHON)

led

Il led è un segnale visivo che indica lo stato di funzionamento del sistema, secondo le seguenti regole:

  • Lampeggia quando il sistema è ENGAGED
  • Spento in tutti gli altri casi
ANALISI: Da richiesta del committente il led si trova sullo stesso nodo del sensor. Come per il sensore, il led fisico deve essere comandto attraverso i GPIO del raspberry, quindi scegliamo di implementare il comando attraverso Python per semplicità d'uso e ampia gamma di librerie. Decidiamo di creare l'attore ledservice, nello stesso conntesto ctx_sensor di sensorservice, per ricevere i comandi da cargoservice per accensione e spegnimento del led. ledservice comunica attraverso mqtt al programma python per comandare il led.

Deployment

Uno dei pregi del linguaggio di modellazione QAK è la possibilità di creare un modello eseguibile del sistema. Al termine di questo sprint è stato infatti realizzato tale modello eseguibile, di cui è possibile effettuare il deployment seguendo questi passi:

  1. Creare e avviare i container di supporto per l'ambiente virtuale del robot con docker compose, definiti nel file robotsmart26.yaml (le istruzioni per il deployment si trovano in fondo al file).
  2. Collegarsi alla pagina web dell'ambiente virtuale, all'indirizzo http://localhost:8085/.
  3. Importare il progetto sprint1 in Eclipse come progetto Gradle.
  4. Lanciare il sistema tramite la Gradle Task run, oppure con il comando gradlew run.
  5. Per eseguire i test, in un prompt dei comandi separato, lanciare il comando gradlew test.

Il sistema appena lanciato resta in stato DISENGAGED. Per osservare un ciclo di movimentazione, lanciare i test.
